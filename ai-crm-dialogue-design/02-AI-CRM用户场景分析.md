# AI-CRM 用户场景分析

> 面向 B2B 销售人员的完整场景梳理、intentType 映射和覆盖度验证。

## 1. 用户画像

**目标用户：B2B 销售人员**

| 特征 | 描述 |
|------|------|
| 技术水平 | 非技术人员，不了解系统内部结构 |
| 说话风格 | 简短口语化，经常省略主语（"录个客户"、"那个商机咋样了"） |
| 使用场景 | 拜访客户前/中/后，需要快速录入和查询 |
| 容错要求 | 高。说错了要能纠正，说不全要能追问 |
| 核心诉求 | 少打字，多办事。能说话解决的不要手动填表 |

## 2. 完整场景树

```
AI-CRM
├── 录入客户
│   └── 客户事前分析
│       └── 企业图谱 [非结构化数据]
│
├── 录入联系人
│
├── 商机录入
│
├── 商机跟进 / 线索跟进
│   ├── 拜访准备（首次/二次等）
│   └── 拜访记录上传
│       └── 客户拜访分析
│           └── 拜访关键内容提取 [非结构化数据]
│
├── 商机6要素分析
│   ├── 需求：客户是否真有痛点/刚需 → 意向产品
│   ├── 预算：有没有钱/预算是否充足 → 预算区间
│   ├── 决策人：能拍板的关键人是否到位
│   ├── 关系：客户信任度/内部支持度 → 图谱关系 [非结构化]
│   ├── 时间节点：明确采购/落地时间
│   └── 竞争：竞品情况/我方优劣势 → 竞争关系 [非结构化]
│
├── 对抗练习（话术模拟训练）
├── 行动建议
├── 普通数据查询
├── 数据分析
└── 问题分析
```

**[非结构化数据] 标注**表示该环节涉及从拜访记录、录音转写等非结构化来源提取信息。

## 3. Skill 分类体系

### 3.1 按系统归属分类

| 分类 | 说明 | 示例 |
|------|------|------|
| 本体 Skill | 基于本体层 YAML 定义的业务操作 | ont.crm.lead_create, ont.crm.opportunity_create |
| 影子系统 Skill | 对接轻云 CRM 的单据操作 | 影子系统客户查询、报价单创建 |
| 外部 Skill | 独立的外部能力 | company-research, pptx |

### 3.2 按数据类型分类

| 分类 | 数据来源 | 处理方式 |
|------|---------|---------|
| 影子系统数据 | 轻云 CRM API | 直接调用 API 查询/写入 |
| 结构化核心数据 | MongoDB + Neo4j | 本体 Skill 通过三库写入 |
| 非结构化核心数据 | 拜访录音/会议纪要/外部研究 | LLM 提取 → 结构化 → 写入 |

### 3.3 按场景分类

| 场景类型 | 说明 | 典型表达 |
|----------|------|---------|
| 普通数据查询 | 查客户/商机/联系人 | "XX客户有几个商机？" |
| 数据分析 | 统计/趋势/对比 | "这个月成交了多少？" |
| 问题分析 | 基于数据的推理判断 | "为什么这个商机推进慢？" |
| 对抗练习 | AI 扮演客户，销售练习话术 | "帮我模拟客户异议处理" |
| 行动建议 | 基于上下文的下一步推荐 | "接下来我该做什么？" |

## 4. 场景与 intentType 映射

### 4.1 7 种 intentType

| intentType | 触发条件 | 处理方式 |
|-----------|---------|---------|
| `execute_skill` | 用户要执行一个明确的操作 | 调用 ability 层 Skill Executor |
| `clarify` | 操作明确但缺必填参数 | 生成结构化澄清卡片 |
| `query` | 用户在问问题 | Query Handler + LLM 生成回答 |
| `composite_task` | 任务涉及多个步骤 | Task Orchestrator DAG 编排 |
| `audio_task` | 用户上传录音 | 保留现有 graph.ts 流程 |
| `chitchat` | 闲聊/感谢/无关内容 | 直接文本回复 |
| `resume_pending` | 用户回复了挂起任务的缺失信息 | 恢复挂起任务，merge 参数后执行 |

### 4.2 全场景映射表

| 场景 | intentType | 说明 |
|------|-----------|------|
| 录入客户 | `execute_skill` / `clarify` | 缺字段时先 clarify |
| 客户事前分析（公司研究） | `execute_skill` | company-research Skill |
| 企业图谱生成 | `execute_skill` | 研究结果写入 Neo4j |
| 录入联系人 | `execute_skill` / `clarify` | 单步 Skill |
| 商机录入 | `execute_skill` / `clarify` | 缺字段时先 clarify |
| 拜访准备 | `composite_task` | 多步 DAG：研究→分析→策略→PPT |
| 拜访记录上传（录音） | `audio_task` | 保留现有 LangGraph 流程 |
| 拜访分析/关键内容提取 | `audio_task` 后续 | audio_task 流程的一部分 |
| **商机6要素分析** | **`execute_skill`** | 专用 Skill，多源聚合+维度打分 |
| 6要素单维度提问 | `query` | "预算够吗？"走 Query Handler |
| 对抗练习 | `execute_skill` | 启动 coaching_mode |
| 行动建议 | `query` / `execute_skill` | 简单推荐走 query，复杂走 Skill |
| 普通数据查询 | `query` | 直接查询 |
| 数据分析 | `query` / `execute_skill` | 简单走 query，复杂走专用 Skill |
| 问题分析 | `query` | LLM 推理 + 上下文 |
| 搜索客户 | `query` | Query Handler + MongoDB |
| 意图切换/挂起恢复 | `resume_pending` | 跨轮次任务持续 |
| 闲聊/感谢 | `chitchat` | 直接文本回复 |

## 5. 特殊场景深度分析

### 5.1 商机6要素分析

商机6要素不是简单的查询，而是**跨数据源的结构化分析**。

**数据来源拓扑**：

```
ont.crm.opportunity_6factors Skill
  ├── 需求 ← 商机字段(意向产品) + 拜访记录(非结构化提取)
  ├── 预算 ← 商机字段(预算区间)
  ├── 决策人 ← 联系人表 + Neo4j 图谱(角色/职级)
  ├── 关系 ← Neo4j 图谱(图谱关系) + 拜访记录(信任度信号)
  ├── 时间节点 ← 商机字段 + 拜访记录(时间线索)
  └── 竞争 ← 外部情报 + 拜访记录(竞品提及)
```

**为什么不是 `query`**：
- 需要从 MongoDB、Neo4j、拜访记录等多个数据源聚合
- 输出是结构化卡片（6维雷达图 + 缺项提示），不是纯文本
- 需要对每个维度打分和标注缺项
- 可能需要生成行动建议（哪个维度薄弱、下一步该做什么）

**但单维度提问走 `query`**：
- "XX商机预算够吗？" → 这是一个简单问题，走 Query Handler 即可

**非结构化数据提取策略**：
- 写入时 ETL：拜访录音完成后提取核心字段（客户名、产品意向、下一步行动）
- 分析时 Lazy：6要素分析请求时对拜访记录做按需提取
- 持久化存储方案后续根据查询性能再定

### 5.2 对抗练习（coaching_mode）

对抗练习是**多轮交互式会话模式**，AI 扮演客户，销售练习应对。

**与其他场景的本质区别**：
- 不是一次性查询或操作
- 需要 AI 持续保持一个"角色"（客户角色）
- 多轮来回，直到用户结束

**处理方案**：
1. 归入 `execute_skill`，调用 `ont.crm.coaching_session` Skill
2. Skill 执行时设置 thread 进入 `coaching_mode`
3. 后续 turn 的 System Prompt 动态注入角色扮演指令
4. Turn Router 在 coaching_mode 下优先将用户输入路由给 coaching Skill
5. 用户说"结束练习"或超时，退出模式

不需要新增 intentType，只需在动态上下文中注入 `activeMode` 状态。

### 5.3 跨轮次任务场景

销售人员经常在对话中途切换话题：

```
Turn 1: "帮我录入客户XX公司"
  → clarify: "请问联系人电话？"

Turn 2: "对了，帮我查一下东港的商机"     ← 意图切换！
  → query: 正常回答东港的商机查询
  → 轻提醒: "另外，XX公司客户录入还差联系人电话，需要继续吗？"

Turn 3: "13800138000"
  → resume_pending: 恢复录入任务，补充电话号码
  → 完成录入
```

**三种任务持续性类型**：

| 类型 | 场景 | 处理方式 |
|------|------|---------|
| A: 即时任务（缺字段） | 录入客户缺电话 | PendingTask 挂起 → 等用户补充 |
| B: 后台任务 | 准备拜访材料 | 后台异步执行 → 完成后通知 |
| C: 录音分析 | 上传录音后多步分析 | LangGraph interrupt → 等用户确认 |

## 6. 覆盖度验证

### 6.1 检查方法

逐一对照场景树中的每个叶子节点，确认是否有对应的 intentType。

### 6.2 验证结果

| 检查项 | 结果 |
|--------|------|
| 所有 CRUD 操作 | `execute_skill` / `clarify` 覆盖 |
| 所有查询/分析 | `query` 覆盖（复杂分析用 `execute_skill`） |
| 多步编排 | `composite_task` 覆盖 |
| 录音处理 | `audio_task` 覆盖 |
| 非结构化数据 | 通过 ETL + Lazy 提取覆盖 |
| 对抗练习 | `execute_skill` + coaching_mode 覆盖 |
| 跨轮次任务 | `resume_pending` + PendingTask + 后台通知覆盖 |
| 闲聊/兜底 | `chitchat` 覆盖 |

### 6.3 结论

**7 个 intentType 覆盖了场景树中列出的全部场景，无遗漏。**

```
execute_skill  → 数据录入、公司研究、6要素分析、对抗练习
clarify        → 缺字段时的参数追问
query          → 数据查询、简单分析、问题推理、行动建议
composite_task → 拜访准备等多步任务
audio_task     → 录音上传与分析
chitchat       → 闲聊/感谢
resume_pending → 跨轮次任务恢复
```
