# AI 原生业务系统 vs 传统低代码：本质差异在哪里？

> 以 CRM 为例，从产品视角解析 AI 原生系统与传统低代码、低代码+AI 增强之间的根本性差异。

---

## 引言

2024-2025 年，几乎所有 SaaS 厂商都在产品里加上了"AI 功能"——AI 摘要、AI 填表、AI 助手侧边栏。但大多数产品的核心架构没有改变：还是表单驱动、菜单导航、字段配置。AI 只是被当作一个"增强插件"嵌入到老架构里。

这种做法的问题不在于它没有用——它确实有用。问题在于它有一个天花板：**AI 能做的事，永远受限于老架构能表达的事。**

AI 原生系统走了一条不同的路：不是在老架构上加 AI，而是以 AI 为前提重新设计系统。本文以 CRM 为例，从产品视角剖析这三种范式的本质差异。

---

## 一、三种范式的定义

### 1. 传统低代码 CRM

代表：Salesforce Classic、Zoho Creator、国内各类低代码 CRM

核心思路：**用配置替代编码，用表单承载业务**。

产品形态：
- 业务对象 = 数据库表
- 功能 = 表单页面 + 列表页面 + 审批流
- 扩展 = 拖拽添加字段 / 配置工作流规则
- 用户操作 = 打开菜单 → 找到页面 → 填写表单 → 提交

### 2. 低代码 + AI 增强 CRM

代表：Salesforce Einstein、HubSpot AI、国内 CRM + AI 助手

核心思路：**保留低代码架构，在关键节点嵌入 AI 能力**。

产品形态：
- 架构不变，还是表单+工作流
- 增加：AI 自动填表、AI 摘要、AI 推荐下一步动作
- 增加：对话侧边栏（问 AI 问题，AI 从系统数据中找答案）
- 用户操作 = 还是菜单 → 表单，但 AI 帮忙填了一些字段

### 3. AI 原生 CRM

核心思路：**以 AI 可理解、可推理、可执行为前提，重新设计系统架构**。

产品形态：
- 对话是主入口，不是"侧边栏"
- 用户说一句话 → 系统理解意图 → 自动选择能力 → 执行并反馈
- 业务对象不只是数据库表，而是有语义定义的本体（AI 能理解其含义和关系）
- 功能不是"页面"，而是"技能"（Skill）——AI 可以自主发现和调用

---

## 二、本质差异：五个维度

### 差异 1: 交互范式——从"人找功能"到"功能找人"

**传统低代码**：

```
用户想录入客户
→ 打开 CRM → 点击"客户管理" → 点击"新建" → 填写 20 个字段 → 提交
```

用户必须知道功能在哪里、表单长什么样、每个字段是什么意思。这是**人适应系统**。

**低代码 + AI 增强**：

```
用户想录入客户
→ 打开 CRM → 点击"客户管理" → 点击"新建" → AI 自动填了 5 个字段 → 用户补充剩余字段 → 提交
```

操作路径没变，只是表单填写更快了。AI 是**提效工具**，不改变交互逻辑。

**AI 原生**：

```
用户想录入客户
→ 对话框说"帮我录入XX公司" → 系统自动识别要创建客户 → 追问缺的字段 → 完成
```

用户不需要知道"客户管理"在哪里，不需要面对 20 个字段的表单。说一句话，系统自己去匹配能力、收集参数、执行操作。这是**系统适应人**。

**本质差异**：传统低代码要求用户掌握系统的"语法"（菜单结构、表单布局）。AI 原生系统理解用户的"自然语言"，把系统语法的翻译工作交给 AI。

### 差异 2: 能力模型——从"页面=功能"到"技能=功能"

**传统低代码**：

功能 = 页面。要新增一个功能，就要新建一个页面（或一套页面：列表页+详情页+编辑页）。

```
新增"拜访准备"功能
→ 设计拜访准备表单 → 配置字段 → 配置列表 → 配置权限 → 发布
```

每个功能都有独立的 UI，用户需要知道去哪个页面操作。

**AI 原生**：

功能 = 技能（Skill）。一个技能是一个**声明式的能力描述**，包含：做什么、什么时候用、需要什么参数。

```
新增"拜访准备"功能
→ 写一个 SKILL.md 文件（描述能力）→ 系统自动注册 → 用户说"帮我准备拜访" → AI 找到并执行
```

技能不需要有自己的 UI 页面——它通过对话界面暴露给用户。新增一个技能，就是新增一个 AI 可以调用的能力，**零 UI 开发成本**。

**本质差异**：低代码的功能单位是"页面"，AI 原生的功能单位是"技能"。页面是给人看的，技能是给 AI 调用的。

### 差异 3: 业务建模——从"数据表"到"语义本体"

**传统低代码**：

业务建模 = 设计数据库表。客户是一张表、商机是一张表、联系人是一张表。表之间通过外键关联。

系统知道"客户表有个字段叫 name"，但不知道"客户是一个有生命周期的业务实体，它可以从线索转化而来，可以关联多个商机，商机有6个关键要素"。

**AI 原生**：

业务建模 = 定义语义本体。不只是定义数据结构，还要定义：

- **对象的含义**：客户是什么？商机是什么？它们的关系是什么？
- **对象的生命周期**：线索 → 客户 → 商机 → 合同，每个阶段的规则是什么？
- **对象上可执行的行为**：创建、更新、分析、评估，每个行为的前提条件和影响是什么？
- **业务规则**：商机金额必须大于 0、客户必须关联至少一个联系人

这些语义定义让 AI 能够"理解"业务，而不只是"存储"数据。当用户说"帮我分析XX商机的健康度"，AI 知道商机有6个要素（需求、预算、决策人、关系、时间节点、竞争），知道该从哪里取数据，知道怎么打分。

**本质差异**：低代码建模面向数据库（让机器存储），AI 原生建模面向语义（让 AI 理解）。

### 差异 4: 扩展方式——从"配置变更"到"能力声明"

**传统低代码**扩展一个功能：

```
1. 在后台添加字段/对象
2. 设计表单页面
3. 配置列表视图
4. 配置工作流/审批流
5. 配置权限
6. 培训用户：新功能在哪、怎么用
```

每一步都需要"管理员"角色操作，涉及 UI 配置、流程配置、权限配置。

**AI 原生**扩展一个功能：

```
1. 写一个 SKILL.md 文件（声明：做什么、什么时候用、需要什么参数）
2. 系统自动注册到 Tool Registry
3. AI 自动能"看到"并调用这个新能力
4. 不需要培训用户——用户只需要用自然语言描述需求
```

**本质差异**：低代码的扩展是"配置驱动"（改配置 → 改 UI → 改流程 → 培训用户），AI 原生的扩展是"声明驱动"（声明能力 → AI 自动集成 → 用户自然使用）。

### 差异 5: 数据利用——从"结构化查询"到"语义化理解"

**传统低代码**：

数据利用 = SQL 查询 + 报表 + 仪表盘。

```
用户想知道商机情况
→ 打开"报表"→ 选择"商机分析"→ 配置筛选条件 → 查看图表
```

系统能告诉你"本月有 50 个商机，总金额 500 万"，但不能告诉你"XX商机推进慢是因为决策人还没接触到"。

**低代码 + AI 增强**：

```
用户想知道商机情况
→ 问 AI 侧边栏："XX商机怎么样了？"
→ AI 从数据库查出基本信息，用自然语言回答
```

AI 能回答结构化数据的问题，但它"看到"的只是数据库字段——看不到拜访记录中的对话内容、看不到客户关系图谱、看不到竞品提及。

**AI 原生**：

```
用户: "帮我分析XX商机的6要素"
→ 系统聚合：商机结构化字段 + 联系人图谱 + 拜访录音中的非结构化信息
→ 6个维度逐一评估打分
→ 输出缺项提示和行动建议
```

AI 原生系统不只查结构化数据——它能从拜访录音中提取"客户提到了竞品A"，从联系人图谱中识别"决策人还没接触到"，从对话历史中推断"客户对预算有顾虑"。

**本质差异**：低代码只能利用结构化数据（数据库字段），AI 原生能理解和利用非结构化数据（录音、对话、文档），并将其转化为可推理的业务洞察。

## 三、一个比喻

如果把 CRM 比作一个办公助理：

- **传统低代码** = 给你一个**文件柜系统**。分类清楚、标签明确，但你得自己走到文件柜前、找到正确的抽屉、拿出正确的表格、填好放回去。
- **低代码 + AI 增强** = 文件柜系统 + 一个**实习生**。实习生能帮你跑腿拿表格、帮你查以前的记录、帮你填一部分内容。但文件柜还是那个文件柜。
- **AI 原生** = 一个**资深助理**。你只需要说"帮我准备明天去XX公司的拜访材料"，助理知道该调研什么公司信息、该准备什么客户分析、该制作什么材料，做完了告诉你结果。你不需要知道文件柜在哪里。

## 四、核心对比表

| 维度 | 传统低代码 | 低代码 + AI 增强 | AI 原生 |
|------|----------|---------------|--------|
| **交互范式** | 菜单→表单→提交 | 同左，AI 辅助填表 | 对话驱动，意图理解 |
| **功能单位** | 页面 (Form/View) | 页面 + AI Widget | 技能 (Skill) |
| **业务建模** | 数据表 + 字段 | 同左 | 语义本体 (AI 可理解) |
| **路由机制** | 菜单导航/URL | 同左 + AI 搜索 | LLM 语义路由 |
| **扩展方式** | 配置页面+流程+权限 | 同左 + 配置 AI 规则 | 声明 SKILL.md，自动可用 |
| **数据利用** | SQL 查询 + 报表 | + AI 问答(结构化数据) | 结构化 + 非结构化语义理解 |
| **用户培训** | 必须培训系统操作 | 需培训，AI 降低难度 | 最小培训，自然语言交互 |
| **复合任务** | 手动多步操作 | 手动 + AI 局部提效 | AI 自动编排多步执行 |
| **跨轮次任务** | 用户自行记住进度 | 同左 | 系统挂起/恢复/后台执行 |
| **新增能力成本** | 页面+流程+权限+培训 | 同左 + AI 规则 | 一个 SKILL.md 文件 |

## 五、"低代码 + AI"的天花板

低代码 + AI 增强是目前市场的主流做法，短期内效果明显（AI 填表、AI 摘要确实提效）。但它有一个结构性天花板：

### 天花板 1: AI 只能在页面粒度上辅助

AI 能帮填表单，但不能帮你决定"该不该填这个表单"。因为 AI 不知道系统有哪些能力、这些能力该在什么时候使用。它只知道当前页面有哪些字段。

### 天花板 2: AI 无法跨页面编排

用户要"准备拜访材料"，涉及查客户信息、做公司研究、生成策略、制作 PPT——这是跨多个"页面"的操作。在低代码架构中，每个页面是独立的，AI 没有能力将它们串联起来。

### 天花板 3: AI 看不到非结构化数据

低代码的数据模型就是数据库表。拜访录音里的竞品信息、客户对话中的关系信号、外部研究报告中的行业洞察——这些非结构化数据不在低代码的数据模型中，AI 无从获取。

### 天花板 4: AI 不理解业务语义

低代码的"客户"是一张有 name/phone/address 字段的表。AI 不知道客户有生命周期、商机有6个评估要素、拜访有前中后三个阶段。缺少语义定义，AI 只能做表面的辅助，无法做深层的业务推理。

**总结**：低代码 + AI 的天花板，本质上是**老架构对 AI 能力的约束**。AI 能做的事被限制在了"表单填写辅助"和"结构化数据问答"的范围内。

## 六、AI 原生的代价

AI 原生不是没有代价。相比低代码 + AI 增强，它面临以下挑战：

| 挑战 | 说明 |
|------|------|
| **架构复杂度高** | 需要语义本体层、技能层、路由层、编排层，比低代码架构复杂得多 |
| **模型依赖** | 核心体验依赖 LLM 的准确性，模型出错 = 系统出错 |
| **可预测性低** | LLM 路由是概率性的，同样的输入不一定得到完全相同的结果 |
| **调试困难** | 用户说"帮我录入客户"但系统理解成了查询，排查链路比表单 Bug 复杂 |
| **建设周期长** | 语义本体的定义、技能的开发、路由的调优都需要深度投入 |
| **企业客户接受度** | 部分企业用户习惯表单操作，对话交互需要转变心智 |

**但这些代价是否值得？取决于一个关键判断**：未来 3-5 年，用户与业务系统的主要交互方式是什么？

如果答案是"还是表单和菜单"，那低代码 + AI 增强足够了。
如果答案是"对话和自然语言"，那 AI 原生是必然方向。

## 七、判断标准：什么业务适合 AI 原生？

不是所有业务系统都适合 AI 原生。以下是判断标准：

| 条件 | 适合 AI 原生 | 适合低代码 + AI |
|------|------------|---------------|
| 用户技术水平 | 低（销售、服务人员） | 高（IT、运维人员） |
| 操作频率 | 高频重复（每天录入/查询） | 低频复杂（年度配置） |
| 非结构化数据 | 大量（录音、邮件、对话） | 少量（主要是结构化字段） |
| 任务复杂度 | 多步编排（拜访准备、商机评估） | 单步操作（填表提交） |
| 业务规则 | 隐性规则多（需推理判断） | 显性规则多（可配置工作流） |
| 个性化需求 | 高（每个客户情况不同） | 低（标准化流程） |

B2B CRM 是一个典型的"适合 AI 原生"场景：用户是非技术的销售人员、每天高频录入和查询、大量非结构化数据（拜访录音、客户对话）、任务涉及多步编排（拜访准备、商机评估）、业务规则多是隐性的（哪个商机健康、哪个客户需要跟进）。

---

## 八、总结

| | 传统低代码 | 低代码 + AI 增强 | AI 原生 |
|--|----------|---------------|--------|
| **本质** | 用配置替代编码 | 用 AI 辅助配置 | 用 AI 替代交互 |
| **AI 的角色** | 不存在 | 插件/增强 | 核心架构 |
| **天花板** | 配置能力的边界 | 老架构对 AI 的约束 | 模型能力的边界 |
| **竞争壁垒** | 配置灵活性 | AI 功能丰富度 | 语义理解深度 |

三者不是简单的代际替代关系。低代码 + AI 增强在很多场景下是务实且有效的选择。但对于特定领域——特别是用户非技术、数据非结构化、任务多步骤的场景——AI 原生系统有结构性优势：

**它不是在老房子上加了一个 AI 天窗，而是以 AI 为地基盖了一栋新房子。**

地基不同，能长出的东西就不同。
