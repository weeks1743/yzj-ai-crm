# AI 原生理念的务实落地：影子系统策略

> 上一篇论述了 AI 原生系统的理想形态。本篇回答一个更现实的问题：如果从零搭建完整本体层的代价太大，如何在保持 AI 原生思想的前提下务实落地？

---

## 引言：理想与现实的张力

05 篇文章描述了一个清晰的图景：AI 原生系统以"语义本体"为核心，让 AI 能理解业务对象的含义、生命周期、行为规则，从而实现对话驱动的智能交互。

但现实中，从零构建一套完整的本体层面临严峻挑战：

- **支撑 10 万用户使用**，需要高可用、高并发、数据隔离的完整基础设施
- **建图/建本体的工作量巨大**，涉及知识图谱、三库同步、数据迁移、版本管理
- **技术栈过于分散**，MongoDB + Neo4j + ChromaDB + SQLite + LLM，任何一环出问题都可能失控
- **投入周期长**，从 YAML 本体定义到三库写入到查询引擎，每一层都是工程量

这不是说本体层的思想是错的——思想完全正确。问题在于**实现路径**：是否一定要自己从零搭建全部基础设施？

答案是：**不一定。思想可以保持不变，实现载体可以替换。**

---

## 一、核心洞察：区分"思想"与"载体"

AI 原生 CRM 的核心思想有三条：

| 思想 | 含义 | 是否依赖自建本体？ |
|------|------|-----------------|
| **语义可理解** | AI 能理解业务对象的含义和关系 | 不依赖。语义可以通过 Skill 描述 + System Prompt 传递给 AI |
| **能力可发现** | AI 能自动发现系统有哪些能力 | 不依赖。Tool Registry 从任何来源注册 Skill 即可 |
| **对话可执行** | 用户说一句话，系统能理解并执行 | 不依赖。Turn Router + Skill Executor 不关心后端是什么 |

**关键认知**：AI 原生的核心不在于"数据存在哪里"或"本体用什么技术实现"，而在于——

1. AI 能看到所有可用能力（Tool Registry）
2. AI 能理解用户意图（Turn Router）
3. AI 能调用正确的能力并执行（Skill Executor）

这三条不依赖于你是否自建了本体层。它们依赖的是**Skill 描述的质量**和**路由决策的准确性**。

---

## 二、影子系统策略：低代码做骨架，AI 做灵魂

### 2.1 什么是影子系统

影子系统是指：**用一个成熟的低代码平台（如轻云/云之家低代码）承载基础的业务数据管理，AI-CRM 系统通过 API 与它交互，把低代码平台当作自己的"数据后端"。**

```
┌──────────────────────────────────────────────┐
│                  系统 A: 轻云 CRM              │
│           (低代码平台 = 影子系统)                │
│                                              │
│  客户管理  │  联系人  │  商机  │  报价  │  合同  │
│  (表单+列表+审批流+权限+报表)                    │
│                                              │
│  ✓ 10万用户高可用     ✓ 数据隔离               │
│  ✓ 权限体系完备       ✓ 审批流/工作流           │
│  ✓ 移动端适配         ✓ 报表/BI                │
└──────────────────┬───────────────────────────┘
                   │ API (CRUD + 查询)
                   │
┌──────────────────┴───────────────────────────┐
│                  系统 B: AI-CRM               │
│           (AI 原生层 = 智能大脑)               │
│                                              │
│  对话入口 → Turn Router → Skill Executor      │
│                                              │
│  AI 独有能力:                                 │
│    ├── 拜访准备 (公司研究→策略→PPT)            │
│    ├── 录音分析 (转写→提取→入库)               │
│    ├── 商机6要素分析 (多源聚合→打分)            │
│    ├── 客户图谱 [非结构化数据]                  │
│    ├── 对抗练习 (话术模拟)                     │
│    └── 行动建议 (上下文推理)                    │
│                                              │
│  影子系统 Skill:                              │
│    ├── 客户 CRUD → 调用轻云 API               │
│    ├── 联系人 CRUD → 调用轻云 API             │
│    ├── 商机 CRUD → 调用轻云 API               │
│    └── 数据查询 → 调用轻云 API                │
└──────────────────────────────────────────────┘
```

### 2.2 两个系统各自的职责

| 职责 | 系统 A (轻云 CRM) | 系统 B (AI-CRM) |
|------|------------------|----------------|
| 数据存储 | 客户/联系人/商机/合同等结构化数据 | 非结构化数据（拜访录音、分析结果、图谱） |
| 基础 CRUD | 增删改查所有业务对象 | 通过 API 调用轻云完成 |
| 权限管理 | 完整的角色权限体系 | 依赖轻云的权限 |
| 审批流 | 报价审批、合同审批等 | 不涉及 |
| 报表/BI | 标准报表、仪表盘 | 不涉及 |
| **对话交互** | 不涉及 | AI 对话入口，意图理解 |
| **智能分析** | 不涉及 | 6要素分析、客户画像、行动建议 |
| **非结构化处理** | 不涉及 | 录音转写、内容提取、图谱构建 |
| **多步编排** | 不涉及 | 拜访准备、复合任务 DAG 执行 |
| **语义路由** | 不涉及 | Turn Router + Tool Registry |

**一句话总结**：轻云做"手脚"（存储和执行基础操作），AI-CRM 做"大脑"（理解、分析、编排、对话）。

### 2.3 为什么不是"低代码 + AI 增强"

这里需要区分清楚：影子系统策略 **不等于** 05 篇文章中批评的"低代码 + AI 增强"。

| 维度 | 低代码 + AI 增强 | 影子系统策略 |
|------|---------------|------------|
| 主系统 | 低代码平台是主体，AI 是附加功能 | AI-CRM 是主体，低代码是数据后端 |
| 用户入口 | 用户在低代码平台的表单中操作 | 用户在 AI 对话框中操作 |
| AI 角色 | 辅助填表、生成摘要 | 核心路由、意图理解、任务编排 |
| 能力扩展 | 在低代码平台配置 + 偶尔加 AI 规则 | 写 SKILL.md，AI 自动集成 |
| 非结构化数据 | 无法处理 | AI-CRM 原生支持 |

**关键区别在于"谁是主体"**：低代码 + AI 增强中，低代码是主体，AI 是锦上添花。影子系统策略中，AI-CRM 是主体，低代码是基础设施——就像你的 App 用了 AWS 的数据库，但你的 App 不是"AWS 增强版"。

## 三、Skill 封装：让 AI 看到影子系统的能力

影子系统策略的技术关键是：**把轻云的能力封装成 AI 可调用的 Skill**。

### 3.1 封装方式

轻云的每一类业务操作，都封装成一个标准的 CRMTool（通过 shadow-adapter）：

```
shadow-adapter 工作流程:
1. 读取轻云 CRM 的单据配置（对象有哪些、字段有哪些、必填规则）
2. 为每个操作生成 CRMTool:
   - name: "shadow.customer_create"
   - prompt(): "在轻云 CRM 中创建客户。必填：company_name。可选：industry, region..."
   - inputSchema: { company_name: { type: "string", required: true }, ... }
   - execute(): 调用轻云 API 完成创建
3. 注册到 Tool Registry
4. AI 通过 Turn Router 自动可见、可调用
```

### 3.2 两种封装协议

| 协议 | 适用场景 | 说明 |
|------|---------|------|
| **SKILL.md** | AI-CRM 原生 Skill | 声明式描述文件，由 ability 层 Skill Executor 执行 |
| **MCP (Model Context Protocol)** | 外部系统集成 | 标准化的 Tool 协议，轻云可以直接暴露 MCP Server |

两种方式都能让 AI "看到"并调用轻云的能力。SKILL.md 更适合紧耦合的核心操作（如客户创建需要同时写入 AI-CRM 的图谱），MCP 更适合松耦合的标准操作（如查询报价单列表）。

### 3.3 Skill 分层

```
Tool Registry 中的 Skill 全景:
│
├── 影子系统 Skill (shadow)
│   ├── shadow.customer_create      ← 轻云 API
│   ├── shadow.customer_query       ← 轻云 API
│   ├── shadow.contact_create       ← 轻云 API
│   ├── shadow.opportunity_create   ← 轻云 API
│   ├── shadow.opportunity_update   ← 轻云 API
│   └── shadow.quotation_query      ← 轻云 API
│
├── AI 原生 Skill (ontology)
│   ├── ont.crm.visit_prepare       ← 拜访准备（复合任务）
│   ├── ont.crm.visit_audio_intake  ← 录音分析
│   ├── ont.crm.opportunity_6factors ← 6要素分析
│   ├── ont.crm.coaching_session    ← 对抗练习
│   └── ont.crm.action_suggest      ← 行动建议
│
└── 外部 Skill (external)
    ├── company-research-pm          ← 公司研究
    └── pptx                         ← PPT 生成
```

AI 不区分 Skill 来自哪里。Turn Router 看到的是统一的工具列表，它只关心"哪个工具能完成用户的意图"。

## 四、语义不丢失：影子系统如何保持 AI 原生的语义能力

最大的疑问是：本体层的核心价值是"语义可理解"——放弃自建本体，语义能力会不会丢？

**不会**，因为语义通过三个层面传递给 AI：

### 4.1 Skill prompt() 传递操作语义

每个 shadow Skill 的 `prompt()` 方法包含了操作的语义描述：

```
"创建客户 (shadow.customer_create)。
 当用户要新建客户、录入公司信息时使用。
 必填：company_name（公司名称）。
 可选：industry（行业）, region（地区）, source（来源）。
 创建后自动关联当前用户为负责人。"
```

AI 通过阅读这段描述就知道"创建客户"是什么、什么时候该用、需要什么参数。

### 4.2 System Prompt 传递业务语义

Turn Router 的静态 System Prompt 中注入业务对象的语义摘要：

```
## 业务对象语义

客户：B2B 企业客户，有行业/地区/规模属性，生命周期为 线索→客户→合作伙伴。
商机：一次销售机会，关联一个客户，有6个关键评估要素（需求、预算、决策人、关系、时间节点、竞争）。
联系人：客户方的具体对接人，有角色（决策人/影响者/使用者）和关系深度。
拜访记录：销售拜访客户的记录，可关联录音，包含结构化摘要和非结构化原文。
```

这段语义描述不依赖于数据存在 MongoDB 还是轻云——它是独立于存储层的业务知识。

### 4.3 非结构化数据层传递深度语义

AI-CRM 独有的非结构化数据（图谱、拜访分析、6要素评分）存储在自己的数据层中，这些才是"深度语义"的载体：

```
语义深度层级:
│
├── 第1层: 字段语义 (轻云提供)
│   "客户名=XX公司, 行业=制造业, 商机金额=50万"
│
├── 第2层: 关系语义 (AI-CRM 图谱)
│   "张三是XX公司的CTO，是本次商机的技术决策人"
│   "XX公司与竞品A有现有合作关系"
│
└── 第3层: 推理语义 (AI-CRM 分析)
│   "该商机6要素中'决策人'维度薄弱——CTO虽已接触但信任度低"
│   "近3次拜访中客户反复提到预算审批流程，时间节点可能延后"
```

轻云提供第1层（结构化字段），AI-CRM 提供第2层和第3层（关系图谱和智能分析）。AI 原生的语义深度来自后两层，不受影子系统策略影响。

## 五、场景走查：影子系统策略下的完整体验

### 场景 1: 录入客户

```
用户: "帮我录入XX公司，制造业的"

Turn Router → execute_skill
  → shadow.customer_create (影子系统 Skill)
  → 调用轻云 API 创建客户
  → 同时触发: ont.crm.company_research (AI 原生 Skill)
     → 外部研究 → 企业图谱写入 AI-CRM 自有存储
  → 返回: "XX公司已创建。公司研究正在后台进行，完成后通知你。"
```

**轻云负责**：客户基础数据的存储、权限、字段校验。
**AI-CRM 负责**：对话理解、参数提取、触发公司研究、图谱构建。

### 场景 2: 商机6要素分析

```
用户: "帮我分析XX商机的6要素"

Turn Router → execute_skill
  → ont.crm.opportunity_6factors (AI 原生 Skill)
  → Skill 内部:
     1. 调用轻云 API 获取商机结构化字段 (预算、产品、阶段)
     2. 调用轻云 API 获取关联联系人列表
     3. 查询 AI-CRM 图谱获取关系/竞争数据
     4. 查询 AI-CRM 拜访分析结果 (非结构化提取)
     5. LLM 聚合6维度打分
  → 输出: opportunity-health-card
```

**轻云负责**：提供商机基础数据、联系人列表。
**AI-CRM 负责**：图谱关系、非结构化分析、6维度打分、结构化卡片输出。

### 场景 3: 拜访准备

```
用户: "帮我准备明天去XX公司的拜访"

Turn Router → composite_task
  → Task Orchestrator 创建 DAG:
     Step 1: shadow.customer_query (轻云 API → 客户基础信息)
     Step 2: company-research-pm (外部研究 → 公司最新动态)
     Step 3: ont.crm.visit_history (AI-CRM → 历史拜访分析)
     Step 4: ont.crm.visit_strategy (LLM → 拜访策略生成)
     Step 5: pptx (→ 拜访 One Page 制作)
  → 后台异步执行
  → 完成后通知: "拜访准备已完成。[查看 One Page]"
```

**轻云负责**：提供客户基础信息。
**AI-CRM 负责**：外部研究、历史分析、策略生成、PPT 制作、任务编排。

## 六、演进路径：从影子系统到完整自治

影子系统策略不是终态，而是一条**渐进式的演进路径**：

```
阶段 1: 影子系统 (当前)
  轻云 CRM 承载全部结构化数据
  AI-CRM 专注对话+分析+非结构化
  两系统通过 API 协作

         ↓ 验证 AI 原生体验可行

阶段 2: 混合自治
  高频操作（如录入客户）逐步迁移到 AI-CRM 自有存储
  轻云保留复杂操作（审批流、报表、权限）
  两套数据双向同步

         ↓ 积累足够用户和数据

阶段 3: 完整自治 (可选)
  AI-CRM 自建完整数据层
  轻云退化为"遗留系统适配层"
  或：轻云本身演进为 AI 原生架构
```

**关键原则**：每个阶段的 Skill 接口不变。从阶段 1 到阶段 3，Turn Router、Tool Registry、Skill 描述保持一致——只有 Skill 的 `execute()` 内部实现从"调用轻云 API"变为"直接操作自有数据库"。**上层 AI 逻辑完全不受迁移影响。**

## 七、影子系统策略的利弊

### 收益

| 收益 | 说明 |
|------|------|
| **工程量可控** | 不需要自建高可用数据层、权限体系、审批流，复用轻云成熟能力 |
| **技术风险低** | 轻云已支撑大规模用户，不存在"自建本体层扛不住10万用户"的风险 |
| **上线速度快** | 聚焦 AI 独有能力（对话、分析、编排），基础 CRUD 直接复用 |
| **思想不妥协** | Tool Registry + Turn Router + Skill 描述的架构完全保留，AI 原生体验不打折 |
| **渐进可演进** | Skill 接口层解耦，后续可逐步替换后端实现，不影响上层 |

### 代价

| 代价 | 说明 | 缓解方式 |
|------|------|---------|
| **API 延迟** | 每次操作多一跳网络调用 | 批量查询、缓存、异步预加载 |
| **数据一致性** | 两套系统的数据需要同步 | 以轻云为主数据源，AI-CRM 侧数据为衍生/增强 |
| **功能边界** | 轻云不支持的操作无法通过影子 Skill 实现 | 这类操作本来就该由 AI-CRM 自有 Skill 处理 |
| **依赖外部平台** | 轻云 API 变更/下线的风险 | shadow-adapter 层隔离，API 变更只需改适配器 |

## 八、两个系统的能力边界

```
                    轻云 CRM                    AI-CRM
                    (影子系统)                   (AI 原生层)
                 ┌─────────────┐           ┌─────────────────────┐
  结构化 CRUD    │  ✓ 客户管理  │           │                     │
                 │  ✓ 联系人   │  ← API →  │  shadow Skill 封装   │
                 │  ✓ 商机     │           │                     │
                 │  ✓ 报价/合同 │           │                     │
                 ├─────────────┤           ├─────────────────────┤
  流程/权限      │  ✓ 审批流   │           │                     │
                 │  ✓ 权限体系  │           │                     │
                 │  ✓ 报表 BI  │           │                     │
                 └─────────────┘           ├─────────────────────┤
                                           │  ✓ 对话入口         │
  AI 独有能力                               │  ✓ Turn Router      │
  (轻云做不到)                              │  ✓ 意图理解/澄清     │
                                           │  ✓ 录音分析          │
                                           │  ✓ 商机6要素分析     │
                                           │  ✓ 客户/联系人图谱   │
                                           │  ✓ 拜访准备(复合任务) │
                                           │  ✓ 对抗练习          │
                                           │  ✓ 行动建议          │
                                           │  ✓ 非结构化数据处理   │
                                           │  ✓ 任务编排/持续性    │
                                           └─────────────────────┘
```

**分界线很清晰**：轻云做"结构化数据的增删改查 + 流程权限"，AI-CRM 做"理解、分析、编排、非结构化处理"。两者通过 Skill 层桥接，用户感知的是一个统一的 AI 助手。

## 九、总结

影子系统策略的本质是：**用成熟的低代码平台解决"存储和管理"的基础设施问题，把所有精力集中在 AI 原生的核心能力上。**

```
放弃的是：自建本体层的技术实现（建图、三库同步、高可用基础设施）
保留的是：AI 原生的核心思想（语义可理解、能力可发现、对话可执行）
换来的是：可控的工程量、可预期的上线节奏、可演进的架构
```

这不是妥协，是**务实**。

AI 原生的价值不在于你用什么技术存数据，而在于——当销售人员说"帮我分析XX商机"的时候，系统能不能真的理解他要什么、知道该调什么能力、自动完成并给出有价值的结果。
