# AI-CRM 实现思路与方案对比

> 三种路由方案的详细对比分析，以及最终选择"两阶段 LLM 结构化输出路由"的论证。

## 1. 问题回顾

核心问题：用户在对话栏说一句话，系统如何决定调用哪个能力？

这本质上是一个**路由问题**——从自然语言输入到系统能力的映射。

## 2. 三种候选方案

### 方案 A: Claude Code 式单阶段 tool_use 路由

**原理**：

```
用户输入
    |
    v
+------------------------------------------+
|  LLM (Claude Sonnet/Opus)                |
|  System Prompt 包含 50+ Tool 的 prompt()  |
|  输出: tool_use function calling 块       |
+------------------------------------------+
    |
    v
直接执行被选中的 Tool
```

LLM 阅读所有工具描述，通过 function calling 直接选择要调用的工具。没有中间的分类步骤，工具选择就是路由。

**优势**：

| 优势 | 说明 |
|------|------|
| 极简架构 | 零路由代码，LLM 直接做决策 |
| 自动扩展 | 新增工具只需添加 prompt()，无需改路由 |
| 灵活组合 | LLM 可以在一个 turn 里选多个工具并行调用 |
| 久经验证 | Claude Code 日活百万级，验证了该模式的可行性 |

**劣势**：

| 劣势 | 说明 |
|------|------|
| 依赖强模型 | 需要 Claude Sonnet/Opus 级别的模型才能在 50+ 工具中准确选择 |
| 无结构化引导 | 不能生成结构化澄清卡片（表单+预填值），只能文本追问 |
| 审计困难 | 没有显式的 intentType 分类记录，难以审计路由决策 |
| 参数提取不稳定 | 开放式 function calling 在参数提取上不如 Structured Output 稳定 |
| 国产模型兼容性差 | DeepSeek/Qwen 的 tool_use 能力弱于 Claude |

### 方案 B: LangGraph 预定义图路由

**原理**：

```
用户输入
    |
    v
+------------------------------------------+
|  预定义 StateGraph                        |
|  条件边 (conditional edges) 决定路由       |
|  路由函数: determineRoute(state)           |
+------------------------------------------+
    |
    +---> start_recording_task
    +---> resume_recording_task
    +---> answer_query
```

预定义状态图，用条件函数（通常是规则/正则）决定走哪条边。这是我们**现有的实现方式**。

**优势**：

| 优势 | 说明 |
|------|------|
| 流程确定性高 | 路由逻辑完全可预测，适合审计 |
| 可视化调试 | LangGraph 提供可视化状态图 |
| Checkpoint 恢复 | SqliteSaver 支持任务状态持久化和恢复 |
| 不依赖 LLM | 路由不消耗 LLM 调用 |

**劣势**：

| 劣势 | 说明 |
|------|------|
| 扩展成本高 | 每新增一种意图需修改图结构和路由函数 |
| 正则覆盖差 | `looksLikeQuery()` 正则无法处理口语化表达 |
| 无法处理模糊意图 | "帮我处理一下XX"——规则无法判断是录入还是查询 |
| 当前已碰壁 | 现有 3 个路由已经无法满足需求，正是我们要解决的问题 |

### 方案 C: 两阶段 LLM 结构化输出路由（推荐）

**原理**：

```
用户输入
    |
    v
+------------------------------------------+
|  Turn Router (LLM Structured Output)      |
|  System Prompt = 角色 + 工具列表 + 规则     |
|  输出: TurnDecision JSON                  |
|  {                                        |
|    intentType: "execute_skill",           |
|    toolCalls: [{ toolName: "...", ... }]  |
|  }                                        |
+------------------------------------------+
    |
    v  (结构化分发)
+---> execute_skill  --> Skill Executor
+---> clarify        --> Clarifier (结构化卡片)
+---> query          --> Query Handler
+---> composite_task --> Task Orchestrator
+---> audio_task     --> 录音流程
+---> chitchat       --> 文本回复
+---> resume_pending --> 恢复挂起任务
```

**第一阶段**：LLM 输出结构化 JSON（TurnDecision），包含 intentType 和对应参数。
**第二阶段**：确定性的代码逻辑根据 intentType 分发到对应处理器。

**优势**：

| 优势 | 说明 |
|------|------|
| 灵活性 | LLM 理解自然语言，能处理口语化、模糊表达 |
| 确定性 | intentType 是枚举值，分发逻辑完全确定 |
| 结构化引导 | clarify 分支可以生成结构化卡片（表单+预填值） |
| 审计友好 | 每次路由决策都有明确的 intentType 记录 |
| 国产模型兼容 | Structured Output (JSON mode) 在 DeepSeek/Qwen 上表现良好 |
| 自动扩展 | 新增 Skill 只需添加 SKILL.md，prompt 自动生成 |
| 可观测 | TurnDecision 可以完整记录和回放 |

**劣势**：

| 劣势 | 说明 |
|------|------|
| 额外延迟 | Turn Router 需要一次 LLM 调用（约 0.5-1 秒） |
| Prompt 设计 | System Prompt 需要精心设计决策规则 |
| 7 分支维护 | intentType 枚举值变更需要同步修改处理器 |

## 3. 多维度对比

| 维度 | 方案 A (单阶段 tool_use) | 方案 B (预定义图) | 方案 C (两阶段结构化输出) |
|------|----------------------|----------------|----------------------|
| **用户画像适配** | 面向开发者（高容错） | 通用 | 面向销售（结构化引导） |
| **路由准确性** | 高（依赖强模型） | 低（正则覆盖不足） | 高（LLM + 决策规则） |
| **审计合规** | 差（无显式分类） | 优（确定性路由） | 优（intentType 记录） |
| **模型要求** | Claude Sonnet/Opus | 无 | DeepSeek/Qwen 即可 |
| **扩展成本** | 零（加工具自动可用） | 高（改图+改路由函数） | 低（加 SKILL.md 即可） |
| **澄清体验** | 文本追问 | 无澄清机制 | 结构化卡片 |
| **多步编排** | LLM 自主决定 | 预定义工作流 | LLM 规划 + 确定性执行 |
| **调试可观测** | 需分析 tool_use 日志 | 状态图可视化 | TurnDecision JSON 回放 |
| **延迟** | 低（一次 LLM 调用） | 最低（无 LLM） | 中（Turn Router 一次 + 处理器可能再调） |
| **任务持续性** | 需自行实现 | Checkpoint 原生支持 | 需自行实现（PendingTask + 后台任务） |

## 4. 为什么选择方案 C

### 4.1 用户画像决定

我们的用户是**销售人员**，不是开发者。他们需要：
- 结构化的表单卡片来补全信息（不是"请告诉我联系人电话"的纯文本追问）
- 明确的状态反馈（不是"我正在处理..."的模糊回复）
- 可预期的交互模式（不是每次都不知道系统会做什么）

方案 A（Claude Code 式）适合开发者——他们习惯文本交互，能理解和处理错误。但销售人员需要更多结构化引导。

### 4.2 模型选择决定

我们使用 DeepSeek/Qwen（国产化合规要求）。这些模型在以下方面表现差异：

| 能力 | Claude Sonnet | DeepSeek/Qwen |
|------|-------------|---------------|
| 开放式 tool_use | 优秀 | 一般 |
| Structured Output (JSON mode) | 优秀 | 良好 |
| 参数提取准确性 | 高 | 中 |

Structured Output 在国产模型上的表现显著优于开放式 tool_use，因此方案 C 比方案 A 更稳定。

### 4.3 审计合规决定

企业 CRM 需要记录每次路由决策：谁在什么时候说了什么话，系统判断为什么意图，调用了什么能力。方案 C 的 TurnDecision JSON 天然满足这个需求。

### 4.4 从方案 B 的教训

现有系统就是方案 B，已经碰壁：
- 3 个硬编码路由无法满足 15+ 种场景
- 正则分类无法处理口语化表达
- 每次加新能力都要改路由代码

方案 C 解决了这些问题：intentType 由 LLM 判断（处理口语化），新增 Skill 自动可用（不改路由代码）。

## 5. 方案 C 相对方案 A 的核心取舍

| 取舍 | 选择 | 代价 |
|------|------|------|
| 结构化引导 vs 极简架构 | 结构化引导 | 多维护一个 Turn Router 模块 |
| 审计友好 vs 自由度 | 审计友好 | intentType 枚举需要维护 |
| 国产模型兼容 vs 强模型能力 | 国产模型 | Structured Output 比 tool_use 多一层抽象 |
| 确定性分发 vs 涌现行为 | 确定性分发 | 失去了 LLM 自主组合多工具的灵活性 |

**最后一条取舍值得展开**：Claude Code 中 LLM 可以在一个 turn 里自主调用多个工具（如同时读文件+搜索代码），这种涌现行为在方案 C 中受限——LLM 只能输出一个 intentType。但对于 CRM 场景，用户一次通常只做一件事，这个限制可以接受。如果确实需要多步操作，走 `composite_task` 分支由 Orchestrator 编排。

## 6. 总结

```
方案 A (Claude Code 式)  → 面向开发者、强模型、极简但无引导
方案 B (预定义图)        → 确定但不灵活，已碰壁
方案 C (两阶段结构化)    → 兼顾灵活与确定，适合销售用户 + 国产模型  ← 推荐
```
