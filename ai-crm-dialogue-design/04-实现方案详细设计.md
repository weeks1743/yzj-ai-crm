# 实现方案详细设计

> 两阶段 LLM 结构化输出路由方案的完整技术设计：8 个核心模块、接口定义、数据模型、流程走查、实施步骤。

## 1. 架构总览

```
用户输入（文本 / 音频 / 卡片回复）
    |
    v
+-------------------------------------------+
|         Turn Router (LLM 自主路由)          |
|  System Prompt = 角色定义                   |
|                + 本体语义摘要               |
|                + Tool Registry 工具列表     |
|                + 线程上下文                 |
|                + 挂起任务摘要               |
|  输出: TurnDecision (structured output)    |
+-------------------------------------------+
    |
    +---> execute_skill  --> Skill Executor (ability 层)
    +---> clarify        --> Clarifier --> 生成澄清卡片
    +---> query          --> Query Handler (LLM + 上下文)
    +---> composite_task --> Task Orchestrator (多步编排)
    +---> audio_task     --> 现有录音流程 (保留)
    +---> chitchat       --> 直接文本回复
    +---> resume_pending --> 恢复挂起任务
```

## 2. 核心模块设计

### 模块 1: Tool 统一接口

**文件**: `chat/server/src/tools/types.ts`

借鉴 Claude Code 的 Tool 接口，核心是 `prompt()` 方法——生成 LLM 可理解的自然语言描述。

```typescript
export interface CRMTool {
  name: string;                              // 唯一标识，如 "ont.crm.lead_create"
  category: 'ontology' | 'external' | 'shadow' | 'query' | 'meta';
  prompt(): string;                          // 核心：LLM 可读的工具描述
  inputSchema: Record<string, unknown>;      // JSON Schema 参数定义
  requiredContext?: string[];                // 执行前提条件
  execute(
    params: Record<string, unknown>,
    ctx: ToolContext
  ): Promise<ToolResult>;
}

export interface ToolContext {
  threadId: string;
  userId: string;
  focusCustomerId?: string;
  focusOpportunityId?: string;
  ontologyId: string;
}

export interface ToolResult {
  success: boolean;
  data?: unknown;
  userMessage?: string;                      // 给用户看的摘要
  sideEffects?: { type: string; detail: string }[];
  error?: string;
}
```

### 模块 2: Tool Registry

**文件**: `chat/server/src/tools/registry.ts`

启动时自动扫描 SKILL.md，注册为 CRMTool。核心方法 `buildToolPrompts()` 将所有可用工具的 prompt 拼接为 LLM System Prompt 的一部分。

**关键职责**:
- `initFromSkillDirs()`: 扫描 `ability/skills/ontology/` 和 `ability/skills/external/` 目录
- `registerShadowTools(config)`: 注册影子系统 Skill（从轻云 API 配置生成）
- `getAvailable(ctx)`: 根据上下文过滤可用工具
- `buildToolPrompts(ctx)`: 构建工具列表文本，注入 System Prompt

### 模块 3: Skill-to-Tool 适配器

**目录**: `chat/server/src/tools/adapters/`

将现有 `ParsedSkill`（ability 层的 skill-core/types.ts）转换为 `CRMTool`。

#### ontology-adapter.ts

从 SKILL.md 的 frontmatter 自动生成 `prompt()`：

```
来源字段: description + when_to_use + arguments + body 中的参数段落
生成格式: "[行为名]。[触发场景描述]。必填：[参数列表]。操作对象：[owner_object]。"

示例输出:
"创建线索 (ont.crm.lead_create)。当用户要新建线索、录入潜在客户时使用。
 必填参数：title(线索标题)。可选：contact_name, contact_phone。
 操作对象：Lead。"
```

`execute()` 实现：调用 `ability-client.ts` 的 HTTP 接口 → ability 层 skill-core executor。

#### shadow-adapter.ts

从轻云 CRM 单据配置生成 CRMTool。`execute()` 调用轻云 API。

#### external-adapter.ts

外部技能（company-research、pptx 等）的 Tool 包装。

### 模块 4: Turn Router

**文件**: `chat/server/src/engine/turn-router.ts`

**替代现有的 `determineRoute()` + `looksLikeQuery()`**

#### TurnDecision 接口

```typescript
export interface TurnDecision {
  intentType: 'execute_skill' | 'clarify' | 'query' | 'composite_task'
            | 'audio_task' | 'chitchat'
            | 'resume_pending';

  // execute_skill 时
  toolCalls?: Array<{
    toolName: string;
    params: Record<string, unknown>;
    confidence: number;
  }>;

  // clarify 时
  clarification?: {
    missingFields: string[];
    questionToAsk: string;
  };

  // composite_task 时
  compositePlan?: Array<{
    stepId: string;
    toolName: string;
    params: Record<string, unknown>;
    dependsOn: string[];
  }>;

  // resume_pending 时
  resumeTarget?: {
    taskId: string;
    params: Record<string, unknown>;
  };

  // query / chitchat 时
  responseText?: string;
}
```

#### System Prompt 分层

**静态层**（跨会话复用，可利用 prompt caching）：

```
你是金蝶云之家 CRM 的 AI 销售助手。
你的用户是 B2B 销售人员，说话简短口语化，经常省略主语。

## 可用工具
{registry.buildToolPrompts(ctx)}

## 业务对象语义摘要
{从本体 YAML 压缩生成：对象 → 关键属性 → 可执行行为 → 核心规则}

## 决策规则
- 用户要录入/创建/新建 → intentType: "execute_skill"
- 信息不足以填满必填参数 → intentType: "clarify"
- 用户在问问题 → intentType: "query"
- 任务涉及多个步骤 → intentType: "composite_task"
- 用户上传录音 → intentType: "audio_task"
- 闲聊/感谢 → intentType: "chitchat"
```

**动态层**（每次请求更新）：

```
## 当前上下文
- 线程焦点客户: {focusCustomerId / customerName}
- 线程焦点商机: {focusOpportunityId}
- 最近对话摘要: {最近 5 轮}

## 挂起的任务
{pendingTasks.map(t =>
  `- [${t.taskId}] ${t.toolName}: 已收集 ${JSON.stringify(t.collectedParams)},
     缺少 ${t.missingParams.join(", ")}, 等待用户输入`
)}

## 后台运行中的任务
{backgroundTasks.map(t =>
  `- [${t.taskId}] ${t.description}: ${t.completedSteps}/${t.totalSteps} 步已完成`
)}
```

### 模块 5: Clarifier

**文件**: `chat/server/src/engine/clarifier.ts`

当 `TurnDecision.intentType === "clarify"` 时：

1. 从 `missingFields` 和目标 Tool 的 `inputSchema` 计算缺失字段
2. 从线程上下文尝试自动预填（如 focusCustomerId）
3. 生成 `clarification-card` 消息（结构化卡片，含预填值和输入框）
4. 同时创建 `PendingTask` 记录
5. 用户回复后，merge 已有参数，重新进入 Turn Router

### 模块 6: Task Orchestrator

**文件**: `chat/server/src/engine/task-orchestrator.ts`

处理复合任务（如"帮我准备拜访材料"）：

1. Turn Router 输出 `compositePlan`（LLM 生成的步骤列表 + 依赖关系）
2. Orchestrator 验证计划合法性（工具是否存在、参数依赖是否可满足）
3. 按 DAG 拓扑序逐步执行，每步结果写入共享 TaskContext
4. 不依赖用户输入的步骤立即后台异步执行
5. 每步完成后推送 `task-status-card`，全部完成推送 `artifact-card`
6. 异常时暂停并询问用户

### 模块 7: Query Handler

**文件**: `chat/server/src/engine/query-handler.ts`

重构现有 `query-engine.ts`，保留上下文收集逻辑，替换硬编码问答：

1. 从 MongoDB / Neo4j / ChromaDB 收集客户上下文（复用 `buildQueryContext` 核心逻辑）
2. 将上下文 + 问题一起发给 LLM 生成自然语言回答
3. 回答中标注数据来源

### 模块 8: 任务持续性管理

#### 核心原则

1. **用户意图永远最高优先**：用户说别的事就先处理别的
2. **任务与对话轮次解耦**：任务有独立生命周期
3. **不阻塞、不强制**：未完成任务是可忽略的通知
4. **状态持久化到最细粒度**：每步结果写入数据库

#### 三类任务

**类型 A: 即时任务（缺字段的单步 Skill）**

```
Turn 1: 用户 "帮我录入客户XX公司"
  → clarify (缺 contact_phone)
  → 创建 PendingTask: { toolName: "ont.crm.lead_create",
      collectedParams: { title: "XX公司" },
      missingParams: ["contact_phone"],
      status: "waiting_input" }
  → 发送 clarification-card

Turn 2a: 用户 "13800138000"
  → LLM 判断这是对 PendingTask 的回答
  → resume_pending → merge 参数 → execute → 完成

Turn 2b: 用户 "XX客户有几个商机？"  (意图切换)
  → LLM 识别为新查询意图
  → query: 正常回答
  → 追加轻提醒: "之前的客户录入还差联系人电话，需要继续吗？"
```

**类型 B: 后台任务（复合任务中不需要用户的部分）**

```
Turn 1: 用户 "帮我准备明天去XX公司的拜访"
  → composite_task → Task Orchestrator 创建任务
  → 不依赖用户的步骤立即后台执行
  → 用户收到: "正在准备拜访材料..."

Turn 2: 用户继续做别的事（后台任务不阻塞）

Turn 3 (系统主动): 后台任务完成
  → 追加 task-notification: "拜访准备已完成。[查看]"
  → SSE 推送给前端
```

**类型 C: 录音分析任务（保留现有模式）**

保留现有 `graph.ts` 的 LangGraph interrupt 机制，整合到 Turn Router 的上下文中。

#### 通知机制

```typescript
// chat/server/src/engine/notification.ts

function emitTaskNotification(threadId: string, task: ChatTask) {
  // 1. 插入通知消息到线程
  insertMessage({
    threadId,
    role: "assistant",
    kind: "task-notification-card",
    payload: {
      taskId: task.id,
      title: `${task.description} 已完成`,
      status: "success",
      artifacts: task.artifacts,
      actions: ["查看详情", "忽略"],
    }
  });

  // 2. SSE 推送给前端（如果用户在线）
  pushSSEEvent(threadId, { type: "task_completed", taskId: task.id });
}
```

## 5. 关键流程走查

### 流程 1: 录入 → 澄清 → 执行

```
1. 用户: "帮我录入一个客户，XX公司"
2. Turn Router 的 System Prompt 包含工具列表
3. LLM 看到 "ont.crm.lead_create: 创建线索。必填：title"
4. LLM 提取 title="XX公司"，发现缺少 contact_phone
5. TurnDecision = {
     intentType: "clarify",
     clarification: {
       missingFields: ["contact_phone"],
       questionToAsk: "请问联系人电话是？"
     }
   }
6. Clarifier 生成 clarification-card → 前端展示
7. 用户回复 "13800138000"
8. Turn Router merge 参数 → intentType: "execute_skill"
9. 调用 ability-client → skill-core executor → 三库写入
10. 返回成功卡片
```

### 流程 2: 查询

```
1. 用户: "XX客户商机情况怎样？"
2. Turn Router → intentType: "query"
3. Query Handler 收集客户上下文 (MongoDB 商机 + Neo4j 图谱)
4. LLM 基于上下文生成自然语言回答
5. 返回 assistant-text 消息
```

### 流程 3: 复合任务

```
1. 用户: "帮我准备明天去XX公司的拜访材料"
2. Turn Router → intentType: "composite_task"
3. compositePlan = [
     { stepId: "research", toolName: "company-research-pm", dependsOn: [] },
     { stepId: "context", toolName: "meta.query_customer_context", dependsOn: [] },
     { stepId: "strategy", toolName: "meta.visit_strategy", dependsOn: ["research","context"] },
     { stepId: "onepage", toolName: "pptx", dependsOn: ["strategy"] }
   ]
4. Task Orchestrator 按 DAG 执行，step 1+2 并行后台执行
5. 每步推送 task-status-card
6. 完成后推送 artifact-card（下载拜访 One Page）
```

### 流程 4: 跨轮次任务持续

```
Turn 1: 用户 "帮我录入客户XX公司"
  → clarify → 创建 PendingTask (missing: contact_phone)
  → 发送 clarification-card

Turn 2: 用户 "对了，帮我查一下东港的商机"     ← 意图切换
  → Turn Router 上下文有 pendingTask，但 LLM 识别为新查询
  → intentType: "query"，正常回答
  → 追加轻提醒: "另外，XX公司客户录入还差联系人电话，需要继续吗？"

Turn 3: 用户 "13800138000"
  → Turn Router 看到 pendingTask + 用户输入像电话号
  → intentType: "resume_pending"
  → merge 参数 → execute → 完成
```

## 6. 特殊场景设计

### 商机6要素分析

通过专用 Skill `ont.crm.opportunity_6factors` 实现，走 `execute_skill` 分支。

**数据来源拓扑**：

```
Skill 内部执行:
  1. 查询商机结构化字段（预算、时间节点、意向产品）
  2. 查询关联联系人 + Neo4j 图谱（决策人、关系、竞争）
  3. 从拜访记录提取非结构化信号（需求强度、关系深度、竞品线索）
  4. 6维度打分 + 缺项标注 + 行动建议

输出: opportunity-health-card（结构化卡片，6维雷达图 + 缺项提示）
```

### 对抗练习 (coaching_mode)

通过 `ont.crm.coaching_session` Skill 启动，设置 thread 进入 coaching_mode：

1. Skill 执行时设置 thread.activeMode = "coaching"
2. 后续 turn 的 System Prompt 动态注入角色扮演指令
3. Turn Router 在 coaching_mode 下优先将用户输入路由给 coaching Skill
4. 用户说"结束练习"或超时，退出模式

## 9. 关键设计决策汇总

| 决策 | 选择 | 理由 |
|------|------|------|
| 路由机制 | 两阶段 LLM 结构化输出 | 兼顾灵活与确定，适合销售用户 + 国产模型 |
| 工具描述 | 从 SKILL.md 自动生成 | 保持 Skill 定义与对话能力一致 |
| 复合任务 | LLM 规划 + 确定性分步执行 | 企业场景需可追溯、可审计 |
| 任务持续性 | 挂起 + 后台 + 低优先级通知 | 用户意图永远优先 |
| 意图切换 | LLM 自主判断 | pendingTasks 注入上下文，LLM 决策 |
| 录音流程 | 保留现有 graph.ts | 已验证可用 |
| LLM 模型 | DeepSeek / Qwen | 国产化合规 |
