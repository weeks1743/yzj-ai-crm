# 云之家接口映射与待验证清单

## 本篇回答什么问题

本篇回答以下问题：

- `AI销售助手` 设计依赖哪些云之家能力
- 这些能力大致落在哪些官方文档入口
- 哪些接口用途已经明确，哪些还需要联调验证

## 权威文档入口

- 云之家开放平台文档中心：
  <https://open.yunzhijia.com/opendocs/>
- 轻云业务 API：
  <https://open.yunzhijia.com/opendocs/docs.html#/server-api/business/lightcloud>
- 组织 API：
  <https://open.yunzhijia.com/opendocs/docs.html#/server-api/org/index>
- OAuth / Token：
  <https://open.yunzhijia.com/opendocs/docs.html#/server-api/auth/oauth>
- 身份解析：
  <https://open.yunzhijia.com/opendocs/docs.html#/server-api/auth/index>

## 1. 认证与身份解析

### 依赖目的

AI销售助手 需要：

- 获取租户级访问能力
- 解析当前用户身份
- 建立 `eid + appId + userId` 的运行时上下文

### 设计映射

| 能力 | 用途 | 设计依赖 |
|------|------|---------|
| OAuth / Token | 获取调用云之家接口所需 token | 基础设置层、任务系统 |
| 身份解析 | 识别当前登录用户 | 用户上下文、权限与审计 |

### 待验证项

- token 生命周期与刷新机制
- token 是否支持租户 / 应用维度隔离
- 身份解析是否能稳定返回当前用户和租户信息

## 2. 组织同步

> `0.2.0` 当前落地口径：
> 本阶段只接入“查询全部在职人员信息”接口，做手动触发的全量在职人员同步。
> 暂不实现 cron、增量同步、部门同步、角色同步和上下级关系同步。

### 依赖目的

组织同步用于：

- 拉取部门和人员
- 做 AI销售助手 内部人员映射
- 为销售负责人、跟进负责人、权限控制提供基础

### 设计映射

| 能力 | 用途 | 设计依赖 |
|------|------|---------|
| 组织 API | 获取组织、部门、成员信息 | 组织同步任务、身份映射、权限判断 |

### 当前实现映射

- Token：
  `resGroupSecret` 授权级别 `AccessToken`
- 人员全量同步接口：
  `/gateway/openimport/open/person/getall`
- 当前只消费返回中的在职人员字段，不以本阶段交付部门树或角色体系

### 待验证项

- 是否支持首次全量 + 后续增量
- 用户离职 / 部门变更如何反映
- 组织同步与轻云应用内用户字段如何对齐

## 3. 轻云业务对象能力

> `0.6.0` 当前落地口径：
> `admin-api` 已对客户、联系人、商机、商机跟进记录建立“双源模板”链路。
> 当前真实联调范围是 `viewFormDef + getFormByCodeId -> canonical template -> 标准化字段 -> SKILL.md bundle -> searchList/data/list 真实读取 + batchSave 真实写入`。
> 公开 `viewFormDef` 继续承载开放接口侧结构；内部 `getFormByCodeId` 作为字段真相来源，补齐公开接口丢失的 `required`、`edit`、`view`、`readOnly`、`systemDefault`、`placeholder`、`titleEntity` 等信息。
> `references/template-raw.json` 保持兼容路径，但内容已升级为双源合并快照，并附带原始 `sourcePayloads` 与 `mergeDiagnostics`。
> 人员字段当前按云之家人员 `open_id` 参与 skill 输入归一化；关联字段按精确 `formInstId/id` 或完整关系对象输入；`_S_TITLE` 按模板默认标题规则自动派生。

### 依赖目的

轻云业务 API 是影子系统的核心数据来源和回写目标。

它需要支撑：

- 对象元数据读取
- 表单对象实例创建
- 实例查询
- 实例更新
- 关联查询

### 设计映射

| 能力 | 用途 | 设计依赖 |
|------|------|---------|
| 对象 / 模板相关 API | 发现对象与字段结构；公开接口与内部完整模板组合成 canonical template | 动态技能中心 |
| 实例查询 API | 查询客户、联系人、商机、跟进记录 | 记录系统技能、场景技能 |
| 实例创建 API | 创建客户、联系人、商机、跟进记录 | 记录系统技能写回 |
| 实例更新 API | 更新对象字段 | 记录系统技能写回 |

### 本项目必须重点验证的对象

- 客户
- 联系人
- 商机
- 商机跟进记录

### 待验证项

- 真实内部 `getFormByCodeId` HTTP provider 的认证、入参与错误码需要产品化接入
- 公开 `viewFormDef` 与内部 `getFormByCodeId` 的字段差异需要持续记录 `mergeDiagnostics`
- 审批模板中的 `publicOptBoxWidget` 能否稳定带出 `referId`
- 公共选项码表是否应统一走审批接口，还是允许租户先接本地 JSON
- 是否能稳定识别关联对象字段
- 实例查询的过滤、分页、排序能力是否满足动态技能与问答场景
- 实例写回时是否支持幂等或外部请求号

### 双源模板接口分工

| 来源 | 当前 provider | 权责 | 产物位置 |
|------|------|------|------|
| 公开模板 | `viewFormDef` / `PublicTemplateProvider` | 开放接口侧结构、查询/筛选相关配置、公开返回独有 widget 配置 | `sourcePayloads.publicViewFormDef` |
| 内部模板 | 当前仓库 `yzj-api/getFormByCodeId/*.json` / `InternalTemplateProvider` | 字段真相：必填、只读、可编辑、可见、标题规则、系统字段行为 | `sourcePayloads.internalGetFormByCodeId` |
| 合并快照 | `mergeShadowTemplateSnapshots` | 按 `codeId` 合并，保留 provenance 与诊断信息 | `formInfo.widgetMap`、`mergeDiagnostics` |

合并优先级固定为：

- 字段真相优先内部模板
- 开放接口元数据优先公开模板
- 任一侧独有字段不丢弃
- 下游 skill 与 preview/live 只消费 canonical template，不直接猜测单侧接口返回

## 4. 与系统设计的具体关联

### 对基础设置层的关联

- OAuth / Token 配置
- 身份解析配置
- 组织同步配置

### 对影子系统层的关联

- 轻云对象定义读取
- 轻云实例 CRUD
- 双源模板合并诊断与字段 provenance

### 对 AI-CRM 层的关联

- 动态技能生成
- 确认后写回
- 对话查询与场景编排

## 5. 接口验证优先级

### P0：必须先验证

- OAuth / Token 获取
- 身份解析
- 轻云对象元数据读取
- 客户 / 联系人 / 商机 / 跟进记录 的实例查询与写回
- 4 个核心对象的 `create/update` 必填、自动派生、只读阻断回归

### P1：建议尽早验证

- 组织同步
- 关联对象字段读取
- 分页、排序、过滤能力

### P2：后续扩展验证

- 更复杂对象
- 报价 / 订单 / 合同链路

## 6. 对文档集的约束

后续文档凡涉及云之家接口时，应遵循以下约束：

1. 写“用途与依赖”，不在设计稿里硬编码未经验证的参数细节
2. 所有对接口的依赖都应能回指到本篇
3. 未验证部分要明确标成待验证，而不是默认为可用

## 本篇结论

云之家接口不是孤立的外部依赖，而是：

- 身份链路的起点
- 影子系统能力的入口
- 动态技能生成的来源
- AI 写回主数据的出口

因此，认证、身份解析、组织同步、轻云业务 API 的联调优先级必须非常靠前，并且需要被文档化为正式的待验证清单。
