# 计划模板与工具编排原则

## 本篇回答什么问题

本篇回答以下问题：

- 为什么不再把 `scene.*` 定义为一等运行时技能
- 计划模板、工具、策略和 Agent 运行状态分别承担什么职责
- CRM 场景如何作为样例套用，而不是写死到 Agent 核心
- 管理员后台为什么从“场景技能”改为“Agent 治理”

## 0.4.5 结论

自 `0.4.5` 起，系统不再定义“场景技能”作为运行时技能类型。

新的运行链路为：

```text
UserInput -> IntentFrame -> TaskPlan -> Tool Selection -> ExecutionState
```

其中：

- `IntentFrame` 表达用户目标、目标对象、输入材料、约束和缺失信息
- `TaskPlan` 表达推荐步骤、可跳过步骤、确认点和暂停点
- `Tool Registry` 提供实际可调用能力
- `ExecutionState` 记录执行进度、等待状态、失败状态和恢复点

CRM 中的“客户分析”“拜访后整理”“问题陈述”“方案匹配”只作为：

- 计划模板
- slash 入口
- 样例话术
- 验收场景
- 业务域知识包

它们不再是 `scene.*` 运行时技能。

`0.10.3` 起新增的 `yunzhijia-visit-prep` 同样按外部供给能力治理：

- 后台目录技能码为 `ext.yunzhijia_visit_prep`
- Agent 可选工具为 `external.yunzhijia_visit_prep`
- 用户侧显性入口为 `/拜访准备`
- 它可被计划模板引用，但不得注册为 `scene.visit_prepare` 或其他 `scene.*` 运行时技能

## 能力分层

运行时只保留三类工具：

| 类型 | 职责 | 示例 |
|------|------|------|
| `record` | 查询或写入结构化记录对象 | `record.customer.search`、`record.customer.preview_update` |
| `external` | 提供外部处理、研究、生成、导出等能力 | `ext.company_research_pm`、`ext.visit_conversation_understanding`、`ext.yunzhijia_visit_prep` |
| `meta` | 承接 Agent 交互和控制动作 | `meta.clarify_card`、`meta.plan_builder`、`meta.confirm_writeback` |

工具只描述能力边界，不描述业务流程。

`ext.yunzhijia_visit_prep` 的能力边界是“基于已沉淀公司研究 Markdown 与客户初步需求生成拜访准备 Markdown”。是否先查公司研究、缺失时是否等待输入、产物如何保存，属于 Agent 工具编排与策略守卫，不写入公司研究 Markdown 正文。

`0.10.4` 起，`/拜访准备` 的编排规则调整为客户对象优先：先查 `record.customer.search`，再读取 `客户.公司研究材料`。客户关注点只作为可选增强信息；多个客户或多份研究资料命中时，必须使用 Meta Question Card `single_select` 让用户点选。`ext.yunzhijia_visit_prep` 的静态资料沉淀策略为关闭，运行时只返回本轮对话 Markdown，不创建 `analysis_material`、不生成 Evidence Card、不进入向量检索；公司研究与录音相关分析技能继续按各自配置沉淀资料资产。

## 计划模板

计划模板不是工作流，也不是技能。

计划模板只提供：

- 适用意图
- 推荐步骤
- 涉及工具
- 可跳过步骤
- 确认点
- 输出卡片建议
- 证据要求

Agent 可以参考模板生成 `TaskPlan`，但用户可以裁剪、跳过、暂停或稍后继续。

## 策略与确认

确定性策略是 Agent 的硬边界：

- 写主数据必须 preview + confirm
- 高风险字段必须人工确认
- 跨租户引用必须阻断
- 没有证据时不得输出确定性结论
- MVP 禁用的工具必须降级或提示

这些策略不属于某个场景，而是作用于所有工具和计划。

## 管理员后台口径

管理员后台改为“Agent 治理”，固定包含：

- `工具与对象`：Tool Registry、记录对象、外部工具、Meta 工具
- `计划模板`：可推荐的 TaskPlan 模板
- `策略与确认`：写回确认、证据要求、字段风险、跨租户守卫
- `运行观测`：IntentFrame、TaskPlan、ExecutionState、工具调用链和审计结果

系统设置只保留基础接入配置，不再重复管理模型/provider、观测、安全和确认策略。

## 本篇结论

系统应坚持业务无关与奥卡姆剃刀：

1. 不为每个业务场景新增运行时技能类型
2. 不让管理员配置固定工作流
3. 用 `IntentFrame` 承接意图
4. 用 `TaskPlan` 承接计划
5. 用 `Tool Registry` 承接能力
6. 用 `Policy / Guard` 承接安全边界
7. 用 CRM 场景验证平台能力，而不是把 CRM 写死进平台核心
