# yzj-ai-crm 双系统设计文档

## 文档说明

本目录用于沉淀 `yzj-ai-crm` 项目的正式设计方案。

从 `0.1.0` 起，项目默认采用你最新确认的双系统口径：

- `管理员后台`
- `用户 AI 端`

两个入口共享同一业务域、同一租户上下文、同一记录系统对象、同一任务资产与技能治理能力。

`0.0.x` 阶段文档中关于“单产品、AI 主入口、记录系统不是独立产品”的部分，视为早期设计记录；`0.1.0` 当前实现与后续版本治理，以本页和 `iterations/0.1.0/README.md` 的双系统结论为准。

本套文档面向三类读者：

- 产品负责人：快速理解产品边界、核心场景、阶段目标
- 架构与研发负责人：理解系统分层、数据边界、技能体系、对话编排与非结构化数据设计
- 测试与运维负责人：理解可观测性、验收标准、接口映射与待验证项

## 0.1.0 当前基线

### 产品结构

项目当前按同仓库双系统推进：

- 管理员后台：
  基于 `ant-design-pro` 官方工程范式，承接 Dashboard、记录系统治理、AI 资产、技能与编排、系统设置与可观测性
- 用户 AI 端：
  基于 `ant-design/x` 独立式工作台范式，承接会话、任务、录音导入、公司分析、拜访准备等真实业务链路

### 共享底座

两个系统共享以下能力：

- 租户与应用上下文：`eid`、`appId`
- 记录系统对象：
  客户、联系人、商机、商机跟进记录
- 任务与资产：
  会话任务、录音分析资产、公司研究快照、拜访材料结果
- 技能治理：
  Tool Registry、场景技能、外部技能、写回确认策略

### 当前实现目标

`0.1.0` 的目标不是做“看起来像 AI”的单页演示，而是做一版足够正式、足够像产品的高保真原型：

- 后台看起来像真实管理员后台
- AI 端看起来像真实用户工作台
- 数据组织、任务推进、写回确认、审计链路都有正式产品语义

## 当前范围

- 管理员后台：
  Dashboard、记录系统、AI 资产、技能与编排、系统设置
- 用户 AI 端：
  新对话、客户录入与查询、录音导入与拜访分析、公司分析、准备拜访材料、我的任务、历史会话
- 共享数据层：
  `TenantContext`、`ShadowObjectMeta`、`ConversationSession`、`SceneTask`、`ResearchSnapshot`、`AudioImportTask`、`VisitBrief`

## 当前不纳入范围

- 销售报价单 / 销售订单 / 订单变更的 AI 化能力开放
- 合同助手、数字人营销、自动日程推送
- 大规模多 Agent 层级架构
- 图数据库
- 把 PPT 生成为标准输出能力

## 工程实现策略

当前实现策略收敛如下：

- 管理员后台使用 `@umijs/max + @ant-design/pro-components + @ant-design/plots`
- 用户 AI 端使用 `@ant-design/x + @ant-design/x-sdk + @ant-design/x-markdown + antd-style`
- 当前阶段以 mock 数据和 mock provider 为主，但页面结构、路由、容器语义都为真实联调留位
- `3rd/ant-design-pro-master` 与 `3rd/x-main` 仅作为本地参考源码，不直接当成最终运行应用提交

## 当前设计原则

### 1. 双系统，但同一业务域

管理员后台和用户 AI 端是两个正式入口，不再强求单首页或单心智；但两端仍属于同一套业务域和同一套数据能力。

### 2. 记录系统依然是共享底座

客户、联系人、商机、商机跟进记录等结构化主数据仍保留在轻云。AI-CRM 不重复造结构化主数据底座，而是在其上沉淀：

- 对话状态
- 任务状态
- 研究快照
- 录音分析
- 联系人关系
- 实体记忆
- 检索索引

### 3. 强租户隔离

所有持久化数据、对象存储路径、任务、审计、索引、缓存都必须带：

- `eid`
- `appId`

其中 `eid` 是租户隔离核心，`appId` 是应用实例隔离补充维度。

### 4. 场景能力与外部能力必须资产化

录音导入、准备拜访材料等核心场景能力，以及公司分析这类外部能力，都应被设计成可被：

- 准备拜访材料
- 对话查询
- 风险分析
- 后续助手产品

持续复用的能力资产。

## 当前阶段结论

当前文档集给出的阶段性技术建议是：

- v1 数据底座优先使用 `PostgreSQL + pgvector + Object Storage + Redis`
- 同时在数据架构文档中明确对比 `MongoDB + 向量数据库` 路线
- 当前不做最终拍板，但给出清晰的决策门与复核条件

后续如果业务形态明显向“重文档资产平台”演进，再重新评估 Mongo 路线。
