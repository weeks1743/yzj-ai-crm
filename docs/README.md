# AI销售助手重构设计文档

## 文档说明

本目录用于沉淀 `yzj-ai-crm` 项目的重构设计方案。本文档集默认把当前项目视为一个全新重构的 `AI销售助手`，不以任何旧实现为迁移目标，不讨论兼容历史代码，只讨论未来可落地的产品与技术设计。

本套文档面向三类读者：

- 产品负责人：快速理解产品边界、核心场景、阶段目标
- 架构与研发负责人：理解系统分层、数据边界、技能体系、对话编排与非结构化数据设计
- 测试与运维负责人：理解可观测性、验收标准、接口映射与待验证项

## 设计目标

`AI销售助手` 的核心目标不是复制一个传统 CRM 页面系统，而是让销售主要通过对话完成高频任务，同时把结构化主数据沉淀在云之家轻云影子系统中。

本次 v1 设计覆盖以下范围：

- 系统基础设置
- 影子系统层（云之家轻云）
- AI-CRM 层
- 4 个结构化核心对象：
  客户
  联系人
  商机
  商机跟进记录
- 2 个核心场景技能：
  录音导入与拜访分析
  准备拜访材料
- 1 个重要外部技能：
  公司分析（当前按外部技能接入，核心输入为 `companyName`）
- 对话层、任务编排、非结构化数据、测试与可观测性

本次 v1 默认不纳入以下内容：

- 销售报价单 / 销售订单 / 订单变更的 AI 化能力开放
- 合同助手、数字人营销、自动日程推送
- 大规模多 Agent 层级架构
- 图数据库
- 把 PPT 生成为标准输出能力

## 当前实现策略

为尽快把 `AI销售助手` 主链路跑通，当前实现策略收敛如下：

- `录音导入与拜访分析` 优先接入已有的通义 Agent 服务
- 其他外部技能先使用 `mock provider`
- 先把对话、任务、确认写回、资产沉淀这条主链路跑通
- 后续再逐步把 mock provider 替换为真实外部能力

这意味着当前阶段的重点不是一次性把所有外部能力都做完，而是先把：

- 技能注册
- 场景编排
- 任务状态
- 审计与确认
- 数据落库与消费

完整串起来。

## 关键设计原则

### 1. AI销售助手是主入口

用户主要使用 `AI销售助手`，而不是直接使用轻云页面。轻云 `AI销售助手_记录系统` 的角色是结构化主数据系统，不是用户主交互入口。

### 2. 三层逻辑分层，但对外一个产品

系统对外只有一个产品名：`AI销售助手`。

内部逻辑上拆分为：

- 系统基础设置
- 影子系统层
- AI-CRM 层

这个拆分是为了明确职责，不是为了做三个独立系统。

### 3. 影子系统保留主数据

客户、联系人、商机、商机跟进记录等结构化主数据仍保留在轻云。AI-CRM 不重复造结构化主数据底座，而是在其上沉淀：

- 对话状态
- 任务状态
- 研究快照
- 录音分析
- 联系人关系
- 实体记忆
- 检索索引

### 4. 无图数据库

联系人关系、竞争关系、影响关系等一律采用关系型边表表达，不引入图数据库。

### 5. 强租户隔离

所有持久化数据、对象存储路径、任务、审计、索引、缓存都必须带：

- `eid`
- `appId`

其中 `eid` 是租户隔离核心，`appId` 是应用实例隔离补充维度。

### 6. 场景能力与外部能力都必须可复用

录音导入、准备拜访材料等核心场景能力，以及公司分析这类外部能力，都应被设计成可被：

- 准备拜访材料
- 对话查询
- 风险分析
- 后续助手产品

持续复用的能力资产。

## 当前阶段结论

当前文档集给出的阶段性技术建议是：

- v1 数据底座优先使用 `PostgreSQL + pgvector + Object Storage + Redis`
- 同时在数据架构文档中明确对比 `MongoDB + 向量数据库` 路线
- 当前不做最终拍板，但给出清晰的决策门与复核条件

后续如果业务形态明显向“重文档资产平台”演进，再重新评估 Mongo 路线。
