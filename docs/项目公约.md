# 项目公约

## 1. 版本迭代目录

项目根目录必须维护统一的版本迭代目录：

- 根目录固定使用 `iterations/`
- 每次迭代必须创建对应版本子目录
- 每个版本目录内必须包含说明文件 `README.md`

推荐结构如下：

```text
iterations/
  0.0.1/
    README.md
  0.0.2/
    README.md
```

## 2. 版本说明文件要求

每次迭代都必须编写版本说明文件，用于记录本轮版本的设计目标与实际产出。说明文件至少包含：

- 版本号
- 版本目标
- 范围说明
- 关键页面或能力
- 依赖框架
- 验收结果
- 未完成项与下一步计划

## 3. 版本命名规范

版本号统一采用语义化版本格式：

- 小功能：`0.0.1`
- 中型迭代：`0.1.0`
- 大型里程碑：`1.0.0`

版本升级规则：

- 小功能或局部原型迭代，递增最后一位
- 中型能力组合、跨模块原型收口，递增中间位
- 大范围里程碑或正式阶段切换，递增第一位

## 4. 提交与版本映射

每次代码提交都必须能映射到明确版本，不允许出现“没有版本归属”的提交。

提交要求：

- 每次进入新迭代前先创建对应版本目录与说明文件
- 提交信息必须带版本号，例如：`feat(0.0.1): 初始化 AI 主入口原型骨架`
- 对外发布或阶段收口时应创建对应 tag，例如：`v0.0.1`

## 5. 与本项目设计文档的关系

`docs/` 是本项目的正式设计依据。原型、交互、命名、系统分层都必须优先遵循：

- `AI销售助手` 是主入口
- `AI销售助手_记录系统` 是影子系统
- `系统基础设置` 是整套系统的运行根

不允许把本项目实现为“后台管理系统 + AI 页签”的并列双产品结构。
